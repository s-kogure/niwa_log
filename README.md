# niwa.log

# **① プロジェクト概要**

niwa.logは、ハーブガーデンの管理を目的とした個人開発のWebアプリケーションです。植物の状態管理、水やりログの記録、天気情報と連携した水やり必要度のスコアリングを主要機能とします。

### **■ 開発動機と目的**

実用的なガーデニングツールとしての価値に加え、以下の技術的学習目的を含みます。

- ユーザー認証・認可の設計と実装経験（JWT \+ ロールベースアクセス制御）

- マルチユーザーを考慮したデータベース設計（正規化、リレーション設計）

- 外部API連携とキャッシュ戦略（天気予報API）

- プライバシーを考慮した情報設計（位置情報の取り扱い）

# **⑤ 認証・認可設計**

### **■ 認証（Authentication）**

JWTトークンを使用したステートレス認証。「あなたは誰？」を判定するレイヤーです。

- 通常トークン：短期（数時間～1日）

- リメンバートークン：長期（30日TTL）による自動ログイン

- シングルユーザー運用だが、設計上はusersテーブルを用いた汎用構造

### **■ 認可（Authorization）**

ロールベースのアクセス制御。「何をしていい？」を制御するレイヤーです。認証と明確に分離し、ミドルウェアとして独立させます。

| ロール      | 権限                                           | 導入時期 |
| :---------- | :--------------------------------------------- | :------- |
| owner       | 全操作（CRUD、設定変更、ユーザー招待）         | v1       |
| viewer      | 読み取り専用（植物一覧、スコア、制限付き天気） | v2       |
| contributor | 水やりログの追加が可能（将来拡張）             | 将来     |

# **⑥ プライバシー設計**

本アプリは天気予報APIの利用に位置情報を必要とします。これは居住地の推定につながりうるため、プライバシーに配慮した情報設計を採用します。

庭は座標（緯度経度）を直接保持せず、**地域マスタ（regions）への参照のみを持つ**方針です。ピンポイントの座標を保存しないため、居住地特定リスクが構造的に低減します。代表座標（市区町村レベル）は regions 側に内部値として持ち、天気API取得にのみ使用してユーザーには公開しません。

### **■ viewer向け天気表示粒度（任意設定）**

ownerの裁量で viewer への天気表示粒度を段階制御できます。

| レベル | 表示内容                          | 地域特定リスク   |
| :----- | :-------------------------------- | :--------------- |
| 0      | 天気情報なし（スコアのみ）        | なし             |
| 1      | 天気大分類のみ（晴れ/曇り/雨/雪） | 極めて低         |
| 2      | 天気 \+ 暑さランク（5段階）       | 低（季節差あり） |
| 3      | 天気 \+ 気温レンジ（幅表示）      | 中               |

### **■ レスポンス設計例**

| ロール | レスポンスに含むフィールド                                                    |
| :----- | :---------------------------------------------------------------------------- |
| owner  | plants, wateringLogs, weather { condition, temp, humidity, forecast }, scores |
| viewer | plants, wateringLogs, scores, weather { condition } ※設定により変動           |

# **⑦ シーズン管理**

ハーブガーデンは季節性が強く、冬季は大半の植物が休眠期に入ります。この特性をシステム設計に組み込みます。

|                    | アクティブ期（3月～11月目安）          | 休眠期（12月～2月目安）                        |
| :----------------- | :------------------------------------- | :--------------------------------------------- |
| 天気APIポーリング  | 1～2時間ごと                           | 1日×1回                                        |
| 水やりスコアリング | アクティブ（経過時間・天気・種別反映） | 停止（または最小限）                           |
| 気象データ蓄積     | 通常通り                               | 継続（過去データとの比較用）                   |
| 副次的効果         | —                                      | viewerへの天気情報非公開によるプライバシー保護 |

**設計上のポイント：**休眠期の天気データ蓄積は、過去の気象条件と植物の生育状況を紐づけるために有用です。また、休眠期にviewerへの天気情報が非公開となることで、冬季の気温データによる地域特定リスクが自然に解消される副次的効果もあります。

# **⑧ 水やりスコアリング（概念設計）**

水やりの必要度を数値化し、直感的に判断できるようにします。初期バージョンではシンプルな線形モデルを採用します。

### **■ スコア構成要素**

| 要素       | 説明                       | 影響       |
| :--------- | :------------------------- | :--------- |
| 経過時間   | 最後の水やりからの経過時間 | スコア上昇 |
| 種別係数   | 植物の種類ごとの水分要求度 | 係数倍率   |
| 降雨補正   | 直近の降雨実績・予報       | スコア減算 |
| 気温・季節 | 高温時は乾燥が速い         | 係数調整   |

**基本式（v1）：**スコア \= 経過時間 × 種別係数 − 降雨補正

運用しながら重み付けを調整し、必要に応じて複雑化していく想定です。
